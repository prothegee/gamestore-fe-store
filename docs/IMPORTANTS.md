# Important Development Notes & Troubleshooting

This document chronicles critical errors encountered during the development of the GameStore Frontend, their root causes, and the implemented solutions.

---

## 1. Requested and Resolved Page Mismatch
**Why:**
Next.js dynamic routes use brackets (e.g., `[lang]`). During directory creation via the shell, brackets were inadvertently escaped (e.g., `\[lang\]`).

**The Problem:**
Next.js found two conflicting directories: the intended dynamic route `app/[lang]` and a literal directory named `app/\[lang\]`. This caused the Turbopack engine to fail during path resolution, resulting in a 404 or a "page mismatch" error for valid routes like `/en/home`.

**The Solution:**
Definitively removed all escaped directories using `find app -name "*\\\[*" -type d -exec rm -rf {} +` and ensured that only the standard bracketed naming convention is used.

---

## 2. "require is not defined" (Runtime ReferenceError)
**Why:**
A Client Component hook (`useI18n`) was utilized in a page that lacked the `'use client'` directive.

**The Problem:**
Next.js attempted to render the component as a Server Component. The bundling process encountered client-side logic that couldn't be resolved in the server environment, leading to a crash.

**The Solution:**
Added the `'use client'` directive to the top of `about/page.tsx`, `community/page.tsx`, and `support/page.tsx`.

---

## 3. Hydration Attribute Mismatch
**Why:**
Browser extensions (specifically **Dark Reader**) were injecting attributes like `data-darkreader-mode` into the `<html>` and `<svg>` tags before React finished its initial "hydration" phase.

**The Problem:**
React noticed that the HTML generated by the server did not match the HTML present in the browser, causing "Hydration Mismatch" warnings in the console and potential rendering stutters.

**The Solution:**
Added the `suppressHydrationWarning` attribute to the `<html>` tag in `app/layout.tsx` and to specific mobile menu `<svg>` icons.

---

## 4. next/image Unconfigured Hostname
**Why:**
Next.js requires explicit authorization for external domains to prevent unauthorized image optimization attacks.

**The Problem:**
Images from `https://placehold.co` failed to load with a runtime error because the domain was not listed in the project configuration.

**The Solution:**
Updated `next.config.ts` to include `placehold.co` in the `images.remotePatterns` array.

---

## 5. dangerouslyAllowSVG is disabled
**Why:**
By default, Next.js disables SVG support in the `<Image />` component to mitigate potential XSS (Cross-Site Scripting) risks embedded in SVG files.

**The Problem:**
The placeholder images from `placehold.co` were being served as SVGs, causing the build to block them.

**The Solution:**
Enabled `dangerouslyAllowSVG: true` in `next.config.ts` and refactored `dummy-data.ts` to explicitly request `.png` extensions for all placeholder assets.

---

## 6. Blocked Cross-Origin Request (Local Network Testing)
**Why:**
The Next.js development server blocks HMR (Hot Module Replacement) and script requests from origins other than `localhost` for security.

**The Problem:**
When testing the site on a physical mobile device via a local IP (e.g., `192.168.1.59`), the React scripts failed to load. This resulted in a page that *looked* correct but was **non-responsive** (buttons and menus did nothing when tapped).

**The Solution:**
Added the local network IP to the `allowedDevOrigins` configuration in `next.config.ts`.

---

## 7. Mobile UI "Tightness" & Interaction
**Why:**
Original Steam styles utilized fixed large paddings and font sizes that were optimized for desktop displays but felt cramped on mobile.

**The Problem:**
The UI felt "too tight" on small screens, and the lack of an explicit mobile menu made navigation difficult.

**The Solution:**
- Implemented **Responsive Padding** (e.g., `px-4 md:px-8`).
- Created a **Hamburger Menu** with a full-screen drawer for mobile navigation.
- Switched to **Explicit Overlays** for closing menus, ensuring 100% reliability on touch devices where "click outside" logic often fails.

---

## 8. Inconsistent Grid Item Heights
**Why:**
Game images had varying aspect ratios (1:1 and 16:9), and titles had different lengths, causing cards in the store grid to have uneven heights.

**The Problem:**
The grid layout looked broken and unprofessional, with cards within the same row having different dimensions.

**The Solution:**
- Forced a consistent `16:9` aspect ratio on all `GameCard` instances in the `StorePage` grid.
- Implemented `min-h-[2.5rem]` on the title element within `GameCard.tsx`.
- Used `flex flex-col h-full` and `mt-auto` to ensure price metadata always stays at the bottom of the card.

---

## 9. Search and Pagination State Desync
**Why:**
Changing the language or search term while being deep in the paginated results would cause the new results to append to the old ones.

**The Problem:**
Users would see a mix of English and Indonesian titles, or search results that didn't match their latest query, because the `page` state didn't reset on filter changes.

**The Solution:**
Migrated to URL-driven search and pagination (`?q=&page=`). The store page is now a server component that reads `searchParams` directly — no client state, no desync possible. `StoreSearch` pushes to the URL with a 500ms debounce; the server re-fetches and re-renders on each URL change.

---

## 10. Cart Quantity Stale Closure in Batched Calls
**Why:**
After migrating the cart from `localStorage` to server actions with optimistic `setState`, the cart hook captured `cartItems` in a `useCallback` closure. When two `addToCart` calls fired in the same React batch (e.g., inside a single `act()` in tests), both calls saw the same stale `cartItems = []` value and the second call produced `quantity: 1` instead of `2`.

**The Problem:**
The `optimistic` helper computed `next` based on the captured `cartItems` at the time of the `useCallback` definition, not at the time of the `setState` call.

**The Solution:**
Replaced the `optimistic` helper and pre-computed `next` approach with **functional `setState` updates**: `setCartItems(prev => ...)`. Each call in a batch now receives the current state from React's update queue, not the stale closure. The `useCallback` wrapper was removed entirely.

---

## 11. E2E Cart Quantity Race Condition on Hard Navigation
**Why:**
After clicking "Add to Cart", the test immediately called `page.goto('/en/cart')`. This is a hard browser navigation — Next.js re-renders the layout server-side, calling `getCart(lang)` which reads the `cart` cookie. If the `addToCartAction` server action hadn't finished writing the updated cookie yet, the layout read the stale quantity.

**The Problem:**
The test expected `quantity: 2` on the cart page but saw `quantity: 1` because the server-side cookie read raced ahead of the async server action.

**The Solution:**
Changed the test to click the **navbar cart link** instead of calling `page.goto('/en/cart')`. Clicking the link triggers a client-side navigation, which preserves the `CartProvider`'s optimistic state (already updated to `quantity: 2`) without re-reading the server cookie.

---

## 12. E2E Language Dropdown — `<button>` vs `<a>` Selector
**Why:**
The language dropdown in `NavbarClient` renders both language options as `<button>` elements (using `onClick` → `router.push`). The original E2E test used `a:has-text("Indonesian")` which matched zero elements.

**The Problem:**
`page.click('a:has-text("Indonesian")')` timed out because the element is a `<button>`, not an `<a>`.

**The Solution:**
Updated the test selector to `button:has-text("Indonesian")`. Also tightened the language trigger selector from `button:has-text("LANG: en")` to `button:has-text("en")` because "LANG:" is inside a `hidden xs:inline` span and may not be visible at all viewport sizes.
