# Architecture Decision Records (ADR) - GameStore Frontend (20260411_180000)

## Use Next.js 16 App Router for Page Architecture

### Status
Accepted

### Context
The project requires a modern, SEO-friendly, and highly interactive storefront. It must handle complex routing, server-side rendering (SSR), and smooth client-side transitions.

### Decision
We chose **Next.js 16 with the App Router** and **Turbopack**.

### Rationale
- **SSR by Default:** Next.js 16 provides robust Server-Side Rendering out of the box, which is critical for initial load speed and SEO.
- **Dynamic Routing:** The App Router simplifies the implementation of the required `[lang]` dynamic segments.
- **Performance:** Turbopack offers significantly faster hot-reloading during development compared to traditional Webpack setups.

## Implement Language-Prefixed Routing with Server Proxy

### Status
Accepted

### Context
The application must support multiple languages (en, id) and ensure that every path is correctly prefixed (e.g., `/en/home`). Requests to the root or invalid paths must be handled gracefully.

### Decision
We implemented a **Server Proxy (`proxy.ts`)** to manage routing and redirects at the server level.

### Rationale
- **Centralized Logic:** Handling redirects in a single proxy function ensures consistent behavior across the entire site.
- **Language Normalization:** The proxy automatically replaces invalid language codes with the user's preferred language.
- **Persistence:** By managing the `_preferred-lang` cookie at the proxy level, we ensure the user's choice is respected before any HTML is rendered.

## Utilize Classic Steam Dark Theme for Visual Identity

### Status
Accepted

### Context
The application needs a visual identity that is both professional and recognizable. We explored futuristic themes but found they compromised text readability.

### Decision
We decided to implement the **Classic Steam Store Dark Theme** as the primary visual identity.

### Rationale
- **Familiarity:** Users are accustomed to the deep blue and gray palette of the Steam platform.
- **Readability:** Dark backgrounds with light gray and cyan accents provide the highest contrast ratio for metadata.
- **Design Stability:** Avoiding complex translucency reduces "visual noise" and ensures a consistent experience across displays.

## Use Dictionary-Based Localized Content Management

### Status
Accepted

### Context
The application supports English and Bahasa Indonesia. We need to manage both UI labels and dynamic game data across these languages.

### Decision
We implemented a **Dictionary-Based Localization** system using a React Context Provider and centralized translations.

### Rationale
- **Synchronized UI:** The `I18nProvider` ensures that switching the language instantly updates all labels without a full page reload.
- **Data Extensibility:** The data utilities use a similar language-mapped structure, allowing game content to scale alongside UI labels.
- **Ease of Use:** Components can access localized strings through a simple `t()` hook on the client, or `getTranslations(lang)` on the server.

## Adopt Server-First Hybrid Architecture

### Status
Accepted

### Context
The original client-first architecture used `'use client'` on every page, storing session in `localStorage` and fetching data via `useEffect`. This resulted in poor SEO (no server-rendered HTML), a security concern (session exposed to JavaScript), and a slow perceived load (blank page until hydration).

### Decision
We migrated to a **server-first hybrid architecture**: async server components are the default for all pages. `'use client'` is applied only to interactive islands (search input, cart mutations, form validation, navbar dropdowns).

### Rationale
- **Security:** Auth session is managed via an `httpOnly: false` cookie read server-side — not stored in `localStorage`.
- **SEO:** All page content is server-rendered HTML, indexable by crawlers.
- **Performance:** Pages stream from the server; no blank-page hydration phase.
- **Correctness:** URL-driven search/pagination means state is shareable and bookmarkable.

## Manage Shopping Cart via Server Actions and Cookies

### Status
Accepted (supersedes: "Manage Shopping Cart State with LocalStorage and Custom Hooks")

### Context
The original cart implementation used `localStorage` keyed by `cart_{userId}`. This created an inconsistency: auth was server-managed (cookie) but cart was client-managed (localStorage). The server had no visibility into the cart, which prevented server-side checkout rendering and caused inconsistency on hard navigation.

### Decision
We replaced the `localStorage` cart with a **cookie-based cart managed via server actions** (`lib/api/cart.ts`). The `CartProvider` is initialized from the server-read cart on every layout render and applies mutations **optimistically** on the client while calling server actions in the background.

### Rationale
- **Consistency:** Auth and cart are both cookie-based and server-readable. The layout can hydrate both in a single `Promise.all`.
- **Optimistic UI:** Functional `setState(prev => ...)` mutations apply instantly on the client; server actions sync the cookie asynchronously. No loading spinner needed for cart operations.
- **Correctness under batching:** Functional state updates avoid stale-closure bugs when multiple `addToCart` calls are batched in the same React render cycle.
- **Simplicity:** Removes all `useEffect`-based localStorage sync logic, the `cart-update` custom event, and multi-tab synchronization complexity.

## Add Suspense Streaming to Store Page

### Status
Accepted

### Context
The store page previously fetched all 12 games synchronously before sending any HTML. Users on slow connections saw a blank page until the data resolved.

### Decision
We extracted the game grid into a dedicated async server component (`GamesGrid`) and wrapped it in `<Suspense fallback={<GamesGridSkeleton />}>`.

### Rationale
- **Perceived Performance:** The page shell (header, search bar) renders and paints immediately. The game grid streams in as a second chunk.
- **No JS required:** The streaming is server-driven; the skeleton and grid are both server-rendered HTML. No client-side loading state is needed.
- **Modularity:** `GamesGrid` and `GamesGridSkeleton` are self-contained components, making the store page easier to maintain.

## Add Client-Side Validation to Login and Register Forms

### Status
Accepted

### Context
Login and register are server-action forms. Without client-side validation, submitting an empty form incurs a full server round-trip before the user sees an error.

### Decision
We introduced `LoginForm` and `RegisterForm` as `'use client'` wrapper components. They intercept `onSubmit`, validate fields synchronously, display inline errors if invalid, and allow the native form action (server action) to proceed only when all fields are valid.

### Rationale
- **UX:** Instant inline error feedback without a network round-trip.
- **Progressive Enhancement:** The server action still validates on the server; client validation is an additional layer, not a replacement.
- **Minimal scope:** The server page components and server actions are unchanged; validation logic is isolated in the client wrappers.
